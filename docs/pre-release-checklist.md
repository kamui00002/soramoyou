## 0. 必ず実施（全機能共通）

- [ ] **実データでハッピーパスを1回通す**（モック・ビルド成功だけで満足しない）
- [ ] ビルドにエラー・新規警告がない（`build_sim` / Xcode）
- [ ] 関連テストが green（`test_sim`）
- [ ] 既存機能を壊していない（周辺画面を一度触る）

> ⚠️ **「ビルド成功 ≠ 動く」**。コンパイルが通っても、実行時にサーバー通信・初期化・表示で壊れることがある。`build_run_sim` か実機で**実際に動かして目視**する。

## 1. データ / Firestore を触る機能 🔥（今回①の再発防止）

- [ ] **新しいクエリを足したら、必要な複合インデックスを `firestore.indexes.json` に追加して deploy したか**
  - `firebase deploy --only firestore:indexes --project soramoyou-ios`（`.firebaserc` が空なので `--project` 必須）
  - 目安: 「等式フィルタ（`whereField ==`）＋別フィールドの並び替え（`order(by:)`）」は**ほぼ必ず複合インデックスが要る**
- [ ] 書き込み（add/update）だけでなく**読み込み（fetch/list）も実データで成功**するか確認
  - 罠: 書き込みは index 不要で成功するため、**「件数だけ増えて中身が出ない」=読み込み側の index 欠落**を疑う
- [ ] 失敗時にエラーをログ（PostHog / print）に残しているか（クラッシュしない不具合をテレメトリで拾うため）
- [ ] `firestore.rules` の create/update が新フィールドを拒否しないか（`hasOnly` なら許可リストに追加、`hasAll` は追加OK）

## 2. 一覧・表示・著者表示を足す機能 🖼️（今回②の再発防止）

- [ ] **実データを2〜3件入れて、一覧表示まで目視**したか
- [ ] 投稿者・コメント主などの**名前とアイコンが正しく出る**か（内部IDの頭文字など"記号"になっていないか）
- [ ] 表示に必要なデータ（名前・写真）が**正しい置き場から取れているか**
  - 罠: 表示名・写真は **Firestore `users` にしか無い**（Firebase Auth プロフィールは未同期）。`currentUser()` だと nil になる
- [ ] 旧データ（新フィールド無し）でも壊れず表示できるか（後方互換・Optional `?`）

## 3. 蓄積系 MVP 機能（記事「バズ前にやるべき4選」より・後付け不可）

- [ ] **アプリ内フィードバック導線**があるか（不具合・要望をユーザーが送れる）
- [ ] **レビュー誘導**（成功体験の直後・`requestReview()`）— ⚠️**不具合がある状態では出さない**（低評価が溜まる）
- [ ] **行動分析（PostHog）の計装**があるか（主要ファネルの screen_view / 主要操作イベント）
- [ ] UI/UX が最低限**スクショ映えする**仕上がりか

## 4. 提出メカニクス

- [ ] build 番号が**過去提出より大きい**か（重複不可。台帳はメモリ参照）
- [ ] `MARKETING_VERSION` / `CURRENT_PROJECT_VERSION` が意図どおりか（CLI override でなく pbxproj に値をコミット＝ドリフト防止）
- [ ] archive は `ENABLE_USER_SCRIPT_SANDBOXING=YES` のまま通るか（dSYM の inputPaths 修正が有効）
- [ ] 新機能をユーザーに見せるなら **What's New（`WhatsNewContent`）を更新**したか
