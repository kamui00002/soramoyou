# Living Sky 設計書 ⭐️

静止画の空を Metal でループアニメーション化する機能（雲の流れ＋光のゆらぎ→mp4 書き出し）。
SNS 拡散の武器。地上は一切動かさず、空領域だけを自然に動かす。

> 作成: 2026-07-12（設計: Fable 本体／実装: implementer Sonnet 委譲・段階2以降）
> Done条件: ①空だけが自然に動くループ生成 ②継ぎ目のないループ ③mp4書き出し ④SkyMaskProvider 使用 ⑤プレビューがカクつかない

---

## 1. 全体アーキテクチャ（新規ファイルのみ・既存パイプライン不変更）

```
[編集確定後の写真 CIImage]──┐
                            ├─→ LivingSkyEngine.makeFrame(time:) ──→ CIImage（1フレーム）
[SkyMask（1回だけ生成）]────┘        │ 内部: general CIKernel 1パス
                                     ├─→ プレビュー: MTKView + CIRenderDestination（30fps）
                                     └─→ 書き出し: AVAssetWriter → mp4（ループ1周）
```

| 新規ファイル | 置き場所 | 役割 |
|---|---|---|
| `LivingSky.metal` | `Rendering/Shaders/` | general CIKernel 1個（変位＋シマー＋マスク合成を1パス） |
| `LivingSkyEngine.swift` | `Rendering/LivingSky/` | kernel ロード（MetalShaderPipeline と同イディオム: metallib data→CIKernel、失敗時 nil）・`makeFrame(time:) -> CIImage` |
| `LivingSkyParameters.swift` | `Rendering/LivingSky/` | 風向き・速さ・雲量（変位振幅）・シマー強度・ループ長 T・品質モード |
| `LivingSkyPreviewView.swift` | `Views/` | MTKView（30fps）プレビュー |
| `LivingSkyVideoExporter.swift` | `Rendering/LivingSky/` | AVAssetWriter で mp4 書き出し＋Photos 保存 |

**触らないもの**: `FilterGraphBuilder`／`EditRecipe`／`MetalShaderPipeline`／既存27ツール経路。
metal-shader-dev skill の統合チェックリスト 2〜4（FilterGraphBuilder/EditRecipe/EditViewModel 統合）は
**編集ツール向けなので今回は適用外**。適用するのは 1（kernel ロード。ただし自前 Engine 内）と
5（.metal がビルドフェーズに入るか確認）のみ。

⚠️ ただし挙動レベルの例外が1つある: 本機能が pbxproj に追加した
`MTL_COMPILER_FLAGS`/`MTLLINKER_FLAGS = -fcikernel` はターゲット全体に効くため、既存
`ExposureContrast.metal` の CIKernel（露出/明るさ/コントラスト/彩度の1パス）が**本ブランチで初めて
実ロードされる**（従来は同フラグ欠落で常にロード失敗→CIFilterフォールバックで動作していた）。
回帰確認として FilterGraphBuilderTests / EditToolsPhotosParityTests を実行し、失敗2件
（clarity/texture）は本変更前のクリーンビルドでも同一値で失敗する既存不具合であることを
A/B 検証済み（2026-07-12）。

- **入力**は編集確定後の final 画像（既存パイプラインの出力を受け取るだけ）
- **マスク**は `SkyMaskProviderProtocol.makeSkyMask(for:quality:)` を**画像につき1回**生成してキャッシュ
  （フレームごとに再生成しない。プレビュー=`.preview`／書き出し=`.export`）
- **Metal 必須**: kernel ロード失敗時はフォールバックせず機能を非表示（アニメーションに CIFilter 代替は非現実的）

## 2. 動きの数式

### 2.1 雲の流れ — フロー変位＋窓クロスフェード（ループの核・v4）

静止画アニメの定石（Plotagraph 系／ゲームの水面フローマップと同方式）。
写真の空部分を風向きに UV 変位させ続けると伸び切ってしまうため、**複数の位相を交互にリセットしながら
クロスフェード**する。

> **動きモデルの変遷（2026-07-15 実機評価の連鎖）**:
> 段階5までの二相・大振幅版（幅8%/ループ）を実機で確認したところ「雲が動くのではなく**分身**
> みたいになっていて、ループしてる感じもなかった」との評価を受け、**v2（三相クロスフェード＋方向
> 乱流）**へ改訂した。しかし v2 をさらに実機シミュレータで検証すると、T/4差の変化画素が v1 比
> 9.3%→1.28% に激減し動き自体が知覚困難になった。数式で分析すると、三相の正規化ブレンド
> `acc/wsum` は**静止時点でも常に複数コピーが重なって見える「多重露光」構造**を持ち、ユーザー
> 評価は「雲が多重に見えて錯視みたい」だった。並行して検証した**v3（クロスフェードを使わない
> 軌道うねり方式・§2.1末尾参照）**は分身は原理的にゼロだが、ユーザー評価は「ちょっと気持ち悪い」。
> ユーザーは最終的に**ドリフト系（方向性のある流れ）を支持**したため、v2 の多重露光構造を
> **Valve の flow map 定石**（Vlachos, SIGGRAPH 2010「Water Flow in Portal 2」等で広く使われる
> 手法）で解消した **v4「窓クロスフェード＋画素ごと位相オフセット」**を新既定にした。要点は
> ①各位相は大半の時間**単独表示**でスライドし、切替時だけ短くフェードする（多重像が見えるのは
> 各所で周期の約3割の時間だけ）②**位相タイミングを画素ごとにノイズでオフセット**することで、
> 全画面が同時にフェードする「脈動」を排除し、場所ごとにバラバラに雲が世代交代する自然な変化に
> 見せる。

- 位相: `f_1 = frac(t/T + φ(p))`、`f_2 = frac(t/T + φ(p) + 0.5)`（T = ループ長、既定 8 秒。
  `φ(p) = fbm(p·0.003)` が画素ごとの位相オフセット）
- 変位ベクトル: `d(p, f) = A · m(p)^k · F(p) · f`
  - `A` = 最大変位（**幅の最大8%/ループ（speed 1.0時）**。段階3 vision レビューで判明した知覚下限
    （約3px/秒）未満は不可視だったため改定した経緯あり（`LivingSkyParameters.maxDisplacementPx` 参照）
  - `m(p)` = フェザー済みマスク値（0=地上, 1=空）。`^k`（k≈2）で**境界ほど動きを減衰**→地上を引っ張らない
  - `F(p)` = 風向き単位ベクトル × 方向乱流 × 速度ゆらぎ `(1 + speedJitter·(fbm(p·s) − 0.5)·2)`
    （一様な流れに有機的ムラを付与。speedJitter は v2 で 0.3→0.5 に強化・据置。speedJitter=0.5 の場合、
    fbm の実質上限（3オクターブ fbm ≈0.875）で最大約1.375倍まで変位が伸びる）
- **方向の乱流**（分身対策）: 風向きベクトルを場所ごとに粗い fbm で回転する
  `turn(p) = (fbm(p·0.004 + (37.7,37.7)) − 0.5) · 2 · kMaxTurnRad`。全画素が同方向に平行移動すると
  分身がくっきり見えるが、場所ごとに向きを揺らすことでコピー同士が非剛体変形（モーフ）の関係になり
  分身が知覚的に目立たなくなる。回転角 `kMaxTurnRad` は v2 で ±25°(0.44rad) だったが、v4 では
  ドリフト方向の可読性を優先し ±15°(0.26rad) へ緩和した
- **窓クロスフェード（v4: ループの核）**: 三角窓 `tri = 1 − |2·f_1 − 1|` を
  `w_A = smoothstep(0.5−h, 0.5+h, tri)`（h = 0.15）で急峻化し、`c = mix(photo(p−d(p,f_2)), photo(p−d(p,f_1)), w_A)`。
  h=0.15 のとき、各所で全周期の約3割の時間だけ2コピーが重なり、残り約7割は片方のコピーが単独表示
  される（=多重露光の時間を構造的に縮小）

**継ぎ目なしの証明**: `f_1`/`f_2`（位相オフセット込み）はいずれも `frac(t/T + 定数)` の形で
`t/T` の周期関数のため、`frame(0) ≡ frame(T)` が構造的に保証される（v1〜v3 と不変）。

既知のトレードオフ: クロスフェード窓の間（各所で周期の約3割）は依然として2コピーが重なる。
ただし発生タイミングが画素ごとの位相オフセットでバラけるため、v2（三相・全画面同時の重み変化）
のような「静止時点で画面全体が多重露光」には見えない。

**v3: 軌道うねり方式（比較用・motionModel=1）**: クロスフェードを廃し、各画素のサンプル点が
風向きに長軸を向けた閉軌道（楕円）を周回する方式。`ph(p) = 2π·t/T + 2π·fbm(p·0.006)`、
`d(p) = (radius·windDir·cos(ph) + radius·0.35·perp·sin(ph))·m(p)^k`。軌道半径 `radius` は
**常に `A·0.5` の定数ではなく**、`radius = A·kOrbitRadiusRatio(0.5)·(0.5+0.5·fbm(p·0.01+(11.3,11.3)))`
という空間ノイズ変調が掛かる（fbm=1 のとき最大 `A·0.5`、fbm=0 のとき最小 `A·0.25`）。これにより
軌道半径も場所ごとにばらつき、剛体的な同一半径の周回に見えないようにしている。`time01` は
cos/sin の中にのみ入るためループ保証は厳密成立し、合成コピーが複数存在しないため分身は原理的に
ゼロ。位相を空間ノイズでばらつかせることで雲が湧き立つような乱流的な動きに見えるが、ユーザー
評価は「ちょっと気持ち悪い」だったため既定からは外し、`LivingSkyParameters.motionModel = 1` で
選べる比較用として残す。

> **v5: 変位ブレンド（motionModel=0 の実装刷新・2026-07-16）**: v4 は TestFlight 実機評価
> （2026-07-16）で「輪郭の鋭い雲だと分身が見える」との最終NGを受けた。原因は構造的:
> クロスフェード窓の間（周期の約3割）は2コピーが `flow×0.5` ずれて重なったまま**色**をブレンド
> するため、鋭いエッジでは二重線として見えてしまう。**v5 の核心は「色のブレンド」をやめ
> 「サンプル位置（変位）のブレンド」に変えること**: `d = mix(d_2, d_1, w_A)` で変位そのものを
> ブレンドしてから `photo(p−d)` を**1回だけ**サンプルする（v4 は `mix(photo(p−d_2), photo(p−d_1), w_A)`
> と2回サンプルしていた）。`d_1`/`d_2`/`w_A` の定義は v4 と全く同一で time01 の周期関数のままの
> ため、`frame(0) ≡ frame(T)` のループ保証は数式レベルで維持される。常に単一サンプルのため
> 二重像は原理的にゼロになる一方、トレードオフとしてクロスフェード窓の間はサンプル位置が
> 新位相側へ滑り戻る「局所的な巻き戻し」が生じる（巻き戻り速度は順方向の約2.3倍——d は
> 周期関数で1ループの正味移動量はゼロ〔順方向距離=巻き戻し距離〕なので、窓の時間比率
> 〔約3割〕から距離保存で 0.7:0.3 ≈ 2.3倍と導ける）。
> `kPhaseOffsetScale` を 0.003→0.006 に上げてパッチを細かくし、巻き戻りの発生タイミングを
> 場所ごとに分散させることで、全画面が同調して巻き戻るのではなく乱流の渦のように見せている。
> 併せて `LivingSkyParameters.maxDisplacementPx` の係数を 0.08→0.05 に下げ（巻き戻り幅は変位
> 振幅に比例するため）、体感速度の低下を `loopDuration` 既定 8.0→6.0 秒への短縮で補償した。
> motionModel の値自体は増やしておらず、`motionModel=0`（ドリフト）の実装差し替えである。

> **v6: エッジ接線流（edge-tangent flow、motionModel=0 の実装刷新・2026-07-17）**: v5
> （変位ブレンド）はユーザー実機評価（2026-07-17）で「窓の間に順方向の約2.3倍速で巻き戻る」
> 動きが「すごい気持ち悪い」と最終NGを受けた。動きの性格自体（クロスフェードによるドリフト）
> は v4 の時点でユーザー動画判定を通過していたため、v6 は **v4 の色クロスフェードを復活させた
> 上で、v4 がNGになった原因（分身）だけを別の手段で潰す**方針に切り替えた。
>
> 原理: 分身（二重線）が知覚されるのは、被写体の輪郭が「輪郭を横切る方向」に変位するときだけ
> である。輪郭に沿う（接線）方向へ変位すれば、ずれたコピーは元の輪郭の上をなぞるように重なる
> ため二重線にならない。そこで輝度勾配（Sobel 相当の中心差分）で各画素の輪郭の向きを検出し、
> 輪郭が強い場所ではフロー方向を輪郭の接線方向へ smoothstep で滑らかに曲げる
> （`edge = smoothstep(kEdgeLo, kEdgeHi, |∇luminance|)`、`dir = normalize(mix(flow/|flow|, tangent, edge))`）。
> のっぺりした空の内部は輝度勾配が弱く（`edge≈0`）、そもそも分身しようがないため風向きのまま
> 流れる——つまり v6 は「動きの性格は v4 のまま、輪郭付近だけ流れの向きを補正する」パッチである。
> 輝度勾配は元画像 `photo`（時不変）にのみ依存し `time01` を含まないため、`frame(0) ≡ frame(T)`
> のループ保証は数式レベルで不変（`flow` が空間関数のまま、`d_1`/`d_2`/`w_A` は v4/v5 と同一の
> time01 周期関数）。`kEdgeLo`(0.02)/`kEdgeHi`(0.10)（単位=輝度/px）は初期較正値であり、
> 実機目視での調整を前提とする（2026-07-17 v6導入時点では未検証）。`kEdgeAmpReduce`(0.35) は
> 強エッジ付近で変位量自体もわずかに減衰させる安全側パラメータ（輪郭の「すべり」の見えすぎを
> 抑える）。`LivingSkyParameters.maxDisplacementPx` の係数 0.05・`loopDuration` 6.0 秒は
> v5 からそのまま据え置いた（のっぺりした空の内部では v4 と同じ「色クロスフェード」構造の
> ままのため、分身の残余=クロスフェード窓の間の重なり幅 概ね A/2 を小さく保つ意味は引き続き
> あるため）。
>
> **初回較正（2026-07-17）**: シミュレータ実写検証で①雲エッジの粒状ザラつき（テクスチャ由来の
> 局所シア）②電柱・電線の半透明化/分身（前景構造の縦滑り）を確認し、`kGradSamplePx` 2→8px
> （大きな構造の向きだけを拾う）・`edge *= smoothstep(0.6, 0.9, m)`（マスク中間値の前景では
> 偏向を無効化）・`kTangentMax`(0.8) 新設（完全接線化を避けシアを抑制）の3点で対処した。
> `kEdgeLo`/`kEdgeHi` は 0.02/0.10 のまま様子見。

> **v7: レポート準拠ハイブリッド（微ドリフト＋周期モーフ＋二重マスク、motionModel=0 の
> 実装刷新・2026-07-17）**: v6（エッジ接線流）の実機評価に先立ち、技術調査レポート
> （`docs/research/living-sky-research-2026-07.md`）を取得して振り返った結果、v4〜v6 に共通する
> 根本原因は**振幅過大**（幅5%/ループ＝レポート推奨の変位振幅0.4%〜1.8%の4〜8倍）であり、
> ドリフトの大振幅だけで「生きてる感」を出そうとしていたことだったと判明した。v7 は
> レポートの結論に沿って発想を転換する:
>
> 1. **ドリフトを脇役に縮小**: `driftAmplitudePx`（旧 `maxDisplacementPx`。§6参照）の係数を
>    短辺基準0.8%（既定speed=0.5時）へ縮小した。分身の見た目の幅も知覚限界（概ね3px相当）
>    以下に収まるため、v6 のようなエッジ接線偏向による対症療法が不要になる。
> 2. **「生きてる感」の主役を周期モーフと光に**: レポート方式B（周期ノイズによるUVワープ）を
>    円周サンプリング `cyc=(cos(2πt/T), sin(2πt/T))` でノイズ関数の位相に入れることで厳密に
>    周期化した微小モーフ（`microWarpPx`。低周波=雲塊のうねり・高周波=微細な輪郭変化）を追加。
>    控えめな光のゆらぎ（`shimmerAmount` 既定0.012。§6参照）と合わせて動きの説得力を担う。
> 3. **二重マスク＋safeSampleフォールバック**（レポート方式D＋「二重マスク構成」）: 合成用
>    マスク `mask`（compositeMask・従来どおり最終合成 `mix(base,c,m)` に使用）とは別に、
>    侵食＋ブラー済みの変形安全マスク `warpMask` を新設（`LivingSkyEngine.prepare` で1回だけ
>    生成）。変位減衰 `att`（v7のドリフト分岐内でのみ warpMask ベースへシャドウ。v3は従来の
>    `mask` ベースの `att` を継続使用）と、変位後の参照先が安全領域外なら元画素へ戻す
>    safeSample フォールバックの両方に使う。EDT（Euclidean distance transform）は導入せず、
>    warpMask 自体のブラー半径（短辺の0.8%）を距離減衰の近似として使う簡略版。
>
> 二相・中心対称ドリフト `uv = -flow・(p-0.5)・att + micro`（`p=fract(t/T)`, `p2=fract(t/T+0.5)`、
> 三角重み `w1=1-|2p-1|`）は v4〜v6 の「色クロスフェード窓」（`kPhaseOffsetScale`/
> `kXfadeHalfWidth`・画素ごと位相オフセット）を撤去し、レポートの `(p-0.5)` 形式（窓化なしの
> 素の三角重み）に置き換えたもの。`p1`/`p2`/`cyc`/`micro` はいずれも `time01` の周期関数、
> `warpMask` は時不変のため `frame(0)≡frame(T)` のループ保証は数式レベルで不変。v6 のエッジ
> 接線偏向（`kGradSamplePx`/`kEdgeLo`/`kEdgeHi`/`kEdgeAmpReduce`/`kTangentMax`）はパラメータ
> 定義ごと撤去した。v3「軌道うねり」（`motionModel=1`）は不変更（`flowDirPx` の実際の値は
> `driftAmplitudePx` の縮小に伴い小さくなるが、シェーダコードは無変更）。
>
> レポートは MTLTexture 直描き構成（複数パス前処理・毎フレーム1パスの専用レンダラー）を
> 提案しているが、v7 では既存パイプライン変更を避けるため、既存の CIKernel 1パス general
> kernel 構成の中にそのまま移植した（distanceTexture の EDT は用意せず warpMask のブラーで
> 近似する点も含め、レポートの構成そのものではなく簡略移植である点に注意）。

> **v8: タイル化ノイズ雲ベールのオーバーレイ（motionModel=0 の実装刷新・2026-07-17）**: v7
> （微ドリフト＋周期モーフ＋二重マスク）は TestFlight実機評価（2026-07-17）で「振動してるように
> しか見えない」との最終NGを受けた。v4〜v7 はいずれも「写真自身の空をピクセルワープでずらす」
> という同一の設計方針を踏襲しており、振幅・窓・接線偏向・二重マスクとパラメータを変えながら
> 分身/巻き戻り/振動を個別に対症療法してきたが、いずれも根治しなかった。この経緯を受けて
> 第2次 Deep Research（`docs/research/living-sky-research-2026-07-part2-synthesis.md`）を実施し、
> 以下が判明した:
>
> 1. **静止画のピクセルワープによる「写真自身の空の可視ドリフト」は知覚科学的に不可能**——
>    実装7回・数理・知覚科学の三重確認。Braddick (1971) の短距離見かけの運動研究によれば、
>    交互提示の変位が1つの動く物体に融合するのは変位が約5分角までで、15〜20分角で完全に
>    「2枚の別の絵」に見える分身へ崩壊する。iPhone視聴距離換算では融合限界≈画面3pt・完全崩壊
>    ≈20〜30作業pxとなり、二相方式で許される最大ループ移動量から逆算すると「見える速度」と
>    「融合する分身」は原理的に両立不可能（v7 の3px分身が見えなかった一方、v4 の20px分身が
>    くっきり見えたのはこの閾値のちょうど内外にあったため）。
> 2. **商用製品の答えは一貫して「オーバーレイ」**: Motionleap（Lightricks）の自動空アニメは
>    実写雲素材への完全差し替え、Adobe公式のPhotoshopシネマグラフ手順もストックループ動画を
>    マスク領域へ重ねる方式。Lightricksが唯一出荷した変位ワープ論文（SIGGRAPH 2021）も
>    「窓・階段など繰り返し構造」専用で雲・空は対象外と明記されている。
>
> これを受けて v8 は方針を根本転換する: **写真は一切ワープしない**。かわりに、自前生成の
> タイル化ノイズ（格子座標を周期 `kVeilPeriod` で mod する value noise/fbm）による雲ベールを
> 風向きへスクロールし、写真の上に重ねる（レポート「B案：雲オーバーレイ」のB2＝手続き的雲ベール
> による第一試作）。
>
> **仕組み**:
> - 2層パララックス: 層1（大きな雲塊、1ループでタイル周期 `speedPeriods` 回分スクロール）と
>   層2（細かい流れ、層1の2倍速でスクロール・スケール `kVeilScale2` も層1より細かい）を
>   `0.65:0.35` でブレンドする。
> - スクロール座標は風向き単位ベクトル `windDir` を基底にした `u`（風向き軸）・`v`（直交軸）
>   に回転し、`u` 軸だけをスクロールする（風向きに直交する動きは出さない）。
> - 密度→不透明度は `smoothstep(kVeilLo, kVeilHi, veil)` のしきい値変換で「雲の塊」感を出す
>   （ベタ曇りにしない）。
> - 色は `LivingSkyEngine.prepare` が `CIAreaAverage`（`photo×mask` ÷ `mask` の加重平均。
>   `SkyReplacementCompositor.toneMatched` と同型のパターン）で写真の空平均色を1回だけ計測し、
>   明側（white）へ`mix`係数0.55で寄せた値を kernel へ渡す（写真パレット転写）。合成は
>   `c.rgb += (1-c.rgb)*veilRGB*alpha` の screen 系（元の空・雲は下に透けたまま明るく重なる）。
> - 速さスライダーは連続値のままだとループ整合が崩れるため、`speedPeriods ∈ {1, 2, 3}`
>   （整数のみ）へ量子化する。
>
> **ループ整合の証明**: `time01` は elapsed=0 と elapsed=T のどちらでも構造的に 0 になる
> （§既存の time01 計算）ため、`frame(0)≡frame(T)` はこの時点で既に保証されている。より重要な
> 性質は「継続再生時に速度が視覚的に跳ばない」ことで、これは `scrollUnit = fract(time01) ×
> kVeilPeriod × speedPeriods` が `speedPeriods` を整数に量子化しているため、1ループの総移動量が
> 常に `kVeilPeriod` の整数倍になることで保証される。タイル化ノイズ `tileableFbm2` は入力座標を
> `period` の整数倍だけ平行移動しても不変（オクターブごとに座標と周期を連動して2倍にしている
> ため、この不変性はオクターブ間で伝播する）ため、ループの巻き戻り点で場が完全に一致し、
> 速度の跳びが起きない。
>
> **v4〜v7 との構造的な違い**: 写真の参照座標 `photo.sample` は常に元の画素位置 `p` のまま
> （一切ずらさない）ため、分身・局所的巻き戻り・振動・電線/電柱の破綻は**発生源そのものが
> 存在しない**（対症療法ではなく構造的な解消）。`motionModel=1`（v3「軌道うねり」）は比較用に
> 完全に不変更のまま残置している。
>
> **既知の弱点（B1格上げパス）**: タイル化ノイズによる手続き的な雲は、実写の雲と比べて
> 質感が「合成っぽく」見える可能性がある。レポートはこの場合の格上げ先として、CC0系の実写
> ループ雲素材を色転写して重ねる「B1」案を挙げている（§7リスク表参照）。

### 2.2 光のゆらぎ — 円周サンプリングによる周期ノイズ

輝度をノイズでゆっくり揺らす。時間項を**ノイズ空間の円周上**に置くことで周期性を構造的に保証する:

- `η(p, t) = fbm(p·s_shimmer + r·(cos(2πt/T), sin(2πt/T)))` — t が一周すると引数が同一点に戻る＝周期的
- 輝度ゲイン: `L(p,t) = 1 + a_shimmer · m(p) · (η(p,t) − 0.5)`（`a_shimmer` ≈ 0.05、ごく控えめ）
- v1 は輝度のみ（色温度シフトは v2 候補）。マスク乗算済みなので地上の明るさは不変

### 2.3 ノイズ実装

シェーダ内 inline の hash ベース value noise + fbm 2〜3 オクターブ（テクスチャ不要・1パス維持）。
乱数テクスチャ持ち込みは v1 ではしない（`Date.now` 等の非決定要素も不要、完全決定的＝テスト可能）。

### 2.4 最終合成（地上静止の保証）

`c_final = mix(photo(p), c_animated·L(p,t), m_feathered(p))`
`m=0`（地上）なら**元画素そのまま**が数式レベルで保証される。

**マスク再マップ（段階3 vision レビュー指摘#3）**: ヒューリスティックマスクは樹冠・樹間に
0.4〜0.6 の中間値を与えることがあり、`m^k`（k=2）減衰だけでは動きが残って地上がゆらいで見える不具合が
あった。マスクをサンプルした直後に `m = smoothstep(0.45, 0.75, m)` で再マップし、0.45 未満の漏れを
完全にゼロへ落とす（コア空域=0.75以上は1.0のまま、境界のグラデーションは0.45〜0.75帯で維持）。
下限は当初0.30で導入したが、樹冠・樹間の0.4〜0.6帯を通過してしまいゴースト状スミアが残存したため
0.45へ引き上げた（地平線ぎわの晴天域は0.7以上のため上限0.75への引き上げの影響はない）。
早期return判定（m<0.001）はこの再マップ後の値で行う。

## 3. Metal シェーダ構成

- **general CIKernel 1個**に全処理を入れる（GPU 1パス・グラフ最浅）
  - ⚠️ 既存 `ExposureContrast.metal` は**色カーネル**（`sample_t` 受け取り・1画素完結）。
    Living Sky は変位先の画素を読むため **`coreimage::sampler` を受け取る general kernel**。
    テンプレのコピペ不可。`extern "C" float4 livingSky(coreimage::sampler photo, coreimage::sampler mask, float time01, float2 flowDir, float maxDispPx, float shimmerAmp, float speedJitterScale, coreimage::destination dest)` 形
- **ROI コールバックが最重要レビューポイント**: 変位サンプリングするため
  `roiCallback = { _, rect in rect.insetBy(dx: -pad, dy: -pad) }`、
  `pad = maxDispPx·(1 + speedJitter) + roiPaddingMargin(2px)`。速度ムラ
  `flow = turnedDir·(1 + speedJitter·(fbm−0.5)·2)` により実変位は最大
  `(1 + 0.75·speedJitter)·maxDispPx`（speedJitter=0.5 で約1.375倍）まで伸びるため、
  `maxDispPx` ちょうどではなく `(1 + speedJitter)` 倍（同条件で1.5倍・保守的に上回る）を
  基準に ROI を広げる。ExposureContrast の `{ _, rect in rect }` を流用すると**端に未定義画素が出る**
- コンパイル検証: metal-shader-dev skill の `xcrun metal -c -target air64-apple-ios18.0 -fcikernel ...`
- マスクのフェザー: 生成時に一度だけ CIGaussianBlur（**clamp→blur→crop の定石**・SkyReplacementCompositor
  の feather と同様）でソフト化してから kernel に渡す
- **photo は `clampedToExtent()` してから kernel に渡す（段階3 vision レビュー指摘#2）**: 風上側
  （風向き0°なら画像左端）で `p - flow*φ` が extent 外をサンプルすると sampler が透明を返し、
  黒い滲みが常時出る不具合があった。端の画素を外側へ引き伸ばす clamp の定石を適用して解消する
  （`extent:` に渡す出力範囲自体は元の photo.extent のまま変えない）。

## 7. リスクと対策

| リスク | 対策 |
|---|---|
| 雲ベールの質感が「合成っぽく」見える | v8はタイル化ノイズ（手続き的生成）のため実写の雲と質感が異なりうる既知の弱点。格上げ先はB1案（CC0系の実写ループ雲素材を色転写して重ねる。§2.1参照）。vision レビューでの目視判定を要する（2026-07-17 v8導入時点では未検証） |
| 雲ベールの色が写真と合わない | `LivingSkyEngine.computeVeilColor` は `CIAreaAverage` を sRGB 表示値ベースで計測しており厳密なリニア light 空間の加重平均ではない（ガンマ不整合）。white 0.55 mix で大きく寄せているため実用上の影響は小さいと判断したが、色味が想定と異なる場合はここが原因になりうる（vision レビューでの較正対象） |
| ヒューリスティックマスクの誤判定（地上がゆらぐ） | v8は写真を一切ワープしないため「地上が動く」リスク自体は構造的に存在しない。マスク誤判定の残余影響は「地上に雲ベールが薄く乗る/乗らない」の境界のみで、`smoothstep(0.45,0.75)`再マップ（§2.4）で軽減。`SkyMask.confidence` 低時は警告表示 |
| ROI 設定ミスで端にゴミ | v8自体は写真をワープしないためROIパディング不要だが、`motionModel=1`（v3軌道うねり・比較用に残置）は今も変位サンプリングするため、v3のための最小パディング（`driftPx*0.5+roiPaddingMargin`）を維持（LivingSkyEngine.swift参照） |
| 動画の再エンコードで継ぎ目が滲む | 書き出しはキーフレーム間隔をループ長に合わせる（実装時調整） |
| 分身・巻き戻り・振動 | v2〜v7 はいずれも「写真自身の空をピクセルワープでずらす」方針を踏襲し、分身（v2/v4）→巻き戻り（v5）→振動（v7）と対症療法を重ねるも根治しなかった。第2次 Deep Research（`docs/research/living-sky-research-2026-07-part2-synthesis.md`）で「静止画のピクセルワープによる可視ドリフトは知覚科学的に不可能」と確定したため、**v8は写真を一切ワープしない方針へ転換**（§2.1参照）。写真の参照座標が常に元の画素位置のままのため、分身・巻き戻り・振動は発生源そのものが存在しない（対症療法ではなく構造的解消）。v3「軌道うねり」（`motionModel=1`）は比較用に不変更のまま残置（分身原理ゼロだがユーザー評価「気持ち悪い」で既定からは除外） |
| 速さスライダーの量子化が粗い | `speedPeriods ∈ {1,2,3}` の3段階しかないため、速さスライダーを連続的に動かしても体感速度が3段階でしか変化しない場面がある（ループ整合を保つための構造的トレードオフ。§2.1参照） |
